# Schema and Data Rules

## Schema Change Protocol
Before changing any Notion database schema (property type, options, name):

1. **State the current type** and what values existing records hold
2. **State the new type** and how existing values will be treated after the change
3. **Estimate data at risk**: how many records, what values may be lost or misrepresented
4. **Confirm before executing** if any existing data will be silently dropped or altered

Example pre-change declaration:
> "Source.Status is currently `text`. Existing records have values: '已匯入' (12 rows), '' (3 rows).
> Changing to `select` will convert non-empty values to select options if they match; empty values become null.
> No data loss expected. Proceeding."

Never execute a schema change without this declaration.

## Static JSON Data Conflicts
`public/data/cities/*.json` files are generated by GitHub Actions sync from Notion.
Manual edits to these files are always at risk of being overwritten by the next sync.

Rules:
- Prefer fixing data in Notion over fixing it in the JSON file directly
- If a manual JSON fix is necessary, document it as a known divergence in `state/handoffs/latest.md`
- When a merge conflict occurs in these files, always prefer the `origin/main` version
  (it reflects the latest Notion state) UNLESS the conflict involves a deliberate local fix
  that has not yet been synced back to Notion

## Cities Duplicate Prevention
Cities is a reference table but is still subject to duplicates. Known failure mode:

- `ensureCityExists` receives an un-normalized slug (e.g. `大阪` instead of `osaka`)
- `CITY_DATA_MAP["大阪"]` is undefined → fallback city is created with lat=0, lng=0, SortOrder=9999
- The sync script then emits a ghost city entry

Rules:
- `ensureCityExists` must call `normalizeCitySlug()` on its input as the first line (defense in depth)
- Any new write path that creates a City record must also normalize the slug first
- DBA review must include a full Cities dedup check (Name × Slug) — not just Spots and Events

Detection query: look for Cities records where `Slug` matches `CITY_SLUG_MAP` values but Name is duplicated,
OR where `DefaultMapLat = 0` (indicator of a fallback/garbage record).

## Dedup and Upsert Consistency
When adding a new write path (new endpoint, new script) that creates Spot or Event records:
- It must use the same `findExistingRecord` + merge pattern as `confirm-analysis.js`
- It must apply `normalizeName()` for name comparison
- It must use `citySlugMap.js` for slug normalization

Do not introduce a parallel write path that bypasses dedup.
